Paintball Store Web API/Terminal Application

About

This is a web api for a fictionary paintball franchise. I enjoy playing paintball, so I made an api that allows functionalities to get all customers, search for specific customers by email or name, verify their credentials, add a customer, get all stores, get a store's inventory, view the order history of both customer and store, replenish inventory, and place orders. Some specific information about the api include that only managers can replenish the inventory of the store, customers can add multple items to their order, and once they are a registered user, they can order from any store. Each store does have their own list of products and there are checks to make sure you cannot order more than the inventory a store has.

API in Swagger

This is one of the functionalities of the api demonstrated in Swagger (tool for visualising your api). This functionality is for adding a customer to the database. This screenshot shows that AddCustomer requires a reuest body. The information you need to add in this body include the name, address, email and password. The json body has more information than this, but that information is either auto incremented in the database (ex: the customer ID) or does not get read when adding a customer (ex: order info is not read in this case).

Terminal Version

I also have a terminal application for this paintball store. The terminal version was made prior to the web api, so it lacks some of the features from the main api version. In this screenshot, you can see me hitting the menu option for adding a customer. I then have a menu popup for adding a customer. I then enter my first name, last name, and my fake address (even though I am always at this desk). I then enter in an email that does not follow the normal email formatting (used regex for a simple email). Upon pressing enter with the invalid email formatting, I get a message saying that the email entry is invalid and to try again. I added security to this invalid entry feature to that the user has max 3 tries to enter in a field after failing the formatting. It will then tell them that it could not let them register at this time due to too many failed attempts and take them back to the main menu.

Unit Testing

The API version of the project has roughly 52 Unit Tests in C# that tests both the models and the business layer (using moq). I then used sonar cloud to see the code coverage % of my API and to also view any code smells that my API may have.
